# FALSE ACCESS — Learning Rebuild Progress

## Цель

Перестроить обучение так, чтобы игра честно показывала разницу между знакомством с темой, практикой с опорой, самостоятельной работой и подтверждённым выполнением. Игровой прогресс не должен называться профессиональной сертификацией, стажем или готовностью к должности.

## Эталонная база

Программа сверена по структуре и методам оценки с:

- Harvard CS50 Introduction to Cybersecurity;
- Stanford CS155 Computer and Network Security;
- UC Berkeley CS161 Computer Security;
- PortSwigger Web Security Academy;
- NIST NICE Framework Components 2.2.0;
- NIST SP 800-61 Rev. 3;
- OWASP WSTG, API Security и MASVS/MASTG;
- Microsoft SC-200 Security Operations Analyst.

Тексты и задания FALSE ACCESS написаны самостоятельно. Чужие учебники и материалы не копируются целиком.

## Сделано в первой перестройке

### Стартовые этапы 0–3

- все четыре foundation-модуля переписаны;
- добавлены явные evidence, limitations и source mapping;
- убраны production-баллы за стартовые синтетические задания;
- вопросы стали проверять границы выводов, а не только термин;
- итоговые ответы проверяются по многокомпонентной рубрике;
- добавлены минимальная содержательность и запрещённые опасные утверждения;
- команды больше не проходят только по наличию ключевых слов;
- для каждой команды проверяется разрешённый инструмент и полная структура;
- режимы переименованы в «Разбор» и «Проверка»;
- убраны заявления о 100% покрытии и внутренней сертификации.

### Профиль навыков

- `middle-ready` заменён на `lab-readiness` в интерфейсе;
- «Рабочий опыт» переименован в «Симулятор»;
- профиль прямо сообщает, что игровые числа не подтверждают должность;
- curriculum показывает evidence level и ограничение каждого модуля;
- source baseline расширен университетскими курсами и практическими платформами.

### Общие продвинутые модули

- Cloud, Supply Chain, Architecture и Capstone больше не принимают команды по неупорядоченному набору слов;
- проверяется правильный стартовый инструмент и порядок ключевых частей команды;
- hardening patch должен отличаться от исходника, иметь сбалансированные скобки/кавычки и несколько содержательных строк;
- Cloud workload patch исправлен: MFA убрана из machine identity, добавлены OIDC issuer, audience, subject и session duration;
- Kubernetes NetworkPolicy дополнена apiVersion, name, podSelector и явным egress;
- в каждом продвинутом кейсе показывается граница синтетической лаборатории.

### Технические исправления

- DFIR: `fls` и `icat` используют offset NTFS-раздела `2048`;
- DFIR: Chromium timestamp переводится из microseconds since 1601 в Unix epoch;
- Crypto: TLS-проверка требует `-verify_hostname`, `-verify_return_error` и доверенный CA;
- Malware: `ldd` на недоверенном ELF заменён на безопасное чтение `NEEDED` через `objdump -p`;
- проверка Crypto-команд теперь требует правильный инструмент, а не набор anchors в любой строке.

### Аудит

- проверяется наличие письменных рубрик;
- проверяется наличие ограничений и источников;
- проверяется отсутствие production в foundation;
- проверяется структурная валидация команд;
- проверяется отсутствие старых обещаний «сертифицировано», `middle-ready` и «100% покрытия»;
- аудит остаётся внутренним quality gate и не выдаётся за внешний review.

## Что осталось

### Исполняемые лаборатории

- настоящий Python runtime с hidden tests;
- SQLite/PostgreSQL sandbox для SQL;
- локальный уязвимый Web/API стенд;
- shell/PowerShell interpreter с exit code, stderr и permissions;
- настоящие PCAP для network-модулей;
- небольшие EVTX, E01, registry и memory samples для DFIR;
- выполнение KQL, Sigma и YARA на позитивном и чистом corpus.

### Перестройка существующих кейсов

- заменить token/regex-only проверки на parser, runtime или tests;
- переделать варианты ответов, где два неверных ответа очевидно абсурдны;
- заменить готовые отчёты самостоятельным report rubric;
- добавить шум, ложные совпадения, missing telemetry и альтернативные гипотезы;
- добавить последствия ошибочного containment;
- сделать вторые datasets действительно неизвестными, а не переименованной копией;
- проверить остальные команды и конфигурации Windows, Linux, AD, Mail, Mobile и Network по первичной документации.

### Внешняя проверка

- review SOC-аналитика;
- review DFIR-специалиста;
- review AppSec и cloud/security engineer;
- pre-test/post-test на новичках;
- transfer test на новом dataset через неделю после прохождения.

## Текущий честный статус

Стартовая система теперь является структурированным вводным тренажёром. Она лучше проверяет понимание и не выдаёт игровой прогресс за профессию. Полноценным практическим курсом FALSE ACCESS станет после подключения исполняемых лабораторий и внешней проверки.
